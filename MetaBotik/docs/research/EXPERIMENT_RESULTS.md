# Результаты эксперимента: PMR vs Structured Baseline

**Дата:** 15 мая 2026  
**Проект:** MetaBotik — Procedural Meta-Reflection (МКПИ 4.3)  
**Статус:** финальные агрегаты после 3 прогонов (run_02 и run_03 — полный перезапуск LLM)

---

## 1. Цель эксперимента

На одном датасете (10 задач) сравнить два режима промптинга:

| Условие | Промпт | Что измеряем в первую очередь |
|---------|--------|-------------------------------|
| **PMR** | `prompts/system/core_system_prompt.md` (не менялся) | **Процедурная прозрачность** + качество решения |
| **Baseline** | `prompts/baseline_promt.txt` (задача + JSON-схема + CoT) | Качество решения в той же схеме |

**Центральный вопрос статьи:** даёт ли PMR не только «правильный ответ», но и **явный, воспроизводимый алгоритм** его получения?

Для этого введены **две независимые оси метрик**:

1. **Outcome (Zarn)** — насколько ответ **верен** относительно эталона (артефакты, блокеры, policy).
2. **Procedural rigor** — насколько ответ **объясняет процедуру** (тип процедуры, обоснование, альтернативы, критические точки, адаптация, рефлексия).

# Часть A. Procedural Rigor — главный результат эксперимента

## A.1. Что такое procedural rigor простыми словами

**Procedural rigor (процедурная строгость)** — это числовая оценка того, **насколько в ответе явно раскрыта процедура решения**, а не только сам результат.

| | Outcome (Zarn) | Procedural rigor |
|---|----------------|------------------|
| **Вопрос** | «Решение задачи хорошее?» | «Понятно, *как* модель к нему пришла?» |
| **Сверка** | С gold-разметкой (эталон workflow) | Со **структурой PMR-ответа** (поля JSON) |
| **Для PMR** | Может быть высоким | Ожидаемо **высокий** (промпт требует поля) |
| **Для baseline** | Может быть высоким | Ожидаемо **низкий** (другая схема JSON) |

**Аналогия:** два повара выдали одинаково вкусное блюдо (outcome), но один приложил рецепт с обоснованием каждого шага, а второй — только список ингредиентов. Procedural rigor измеряет наличие «рецепта с обоснованием».

## A.2. Откуда берутся числа (пайплайн)

```text
Ответ LLM (JSON на диске)
        ↓
evaluate_procedural_one()   ← src/evaluation/procedural_metrics.py
        ↓
7 подметрик на задачу (0.0 … 1.0)
        ↓
procedural_rigor_score = взвешенная сумма
        ↓
procedural_summary.json + procedural_by_task.jsonl
        ↓
paired_stats (PMR vs baseline по 10 task_id)
```

- **Без вызовов LLM** при оценке — только разбор сохранённого JSON.
- Одна и та же функция для PMR и baseline → сравнение честное.
- Файлы: `results/procedural_summary.json`, `results/procedural_by_task.jsonl` (и то же для `results_baseline/`).

## A.3. Какой JSON ожидает метрика (PMR vs baseline)

### PMR-ответ (оценивается «богато»)

Обязательные блоки для высокого балла:

```json
{
  "procedural_analysis": {
    "problem_classification": "...",
    "selected_procedure": "...",
    "selection_reasoning": "...",
    "alternative_procedures": [
      { "name": "...", "rejection_reason": "..." }
    ]
  },
  "solution_steps": [
    {
      "step": 1,
      "title": "...",
      "action": "...",
      "procedure_logic": "...",
      "critical_points": ["..."],
      "adaptation_notes": ["..."]
    }
  ],
  "reflection": {
    "effectiveness": "...",
    "limitations": ["..."],
    "best_use_cases": ["..."],
    "future_modifications": ["..."]
  }
}
```

### Baseline-ответ (другая схема)

Типичная структура — **без** `procedural_analysis`, `solution_steps`, `reflection`:

```json
{
  "generated_plan": ["шаг 1", "шаг 2", ...],
  "generated_operations": [{ "surface": "Gmail", "operation": "...", "content": "..." }],
  "generated_artifacts": { "workflow_plan": "...", "handoff_summary": "..." },
  "blockers": ["..."]
}
```

План и операции есть, но **нет явного процедурного слоя** → подметрики procedural rigor ≈ 0. Это не баг метрики, а **сигнал**: baseline не обязан объяснять процедуру в формате PMR.

---

## A.4. Семь подметрик: что именно считается

Каждая подметрика — число от **0.0** до **1.0** на одну задачу.

### 1) `procedural_trace_present` (след процедуры)

| | |
|---|---|
| **Смысл** | Есть ли в JSON три блока PMR целиком |
| **1.0 если** | Есть `procedural_analysis` (dict) + непустой `solution_steps` + `reflection` (dict) |
| **0.0 если** | Чего-то из этого нет |
| **PMR** | Почти всегда **1.0** (схема в промпте) |
| **Baseline** | Почти всегда **0.0** (другая схема) |

---

### 2) `procedure_classification` (классификация процедуры)

| | |
|---|---|
| **Смысл** | Назван ли тип процедуры и выбранный метод |
| **1.0** | В `procedural_analysis` заполнены и `problem_classification`, и `selected_procedure` |
| **0.5** | Заполнено только одно поле |
| **0.25–0.5** | Только упоминания типов (linear/branching/…) в тексте JSON, без полей |
| **0.0** | Ничего |

---

### 3) `procedure_justification` (обоснование выбора)

| | |
|---|---|
| **Смысл** | Есть ли причинно-следственное объяснение, *почему* выбрана эта процедура |
| **Источник** | `procedural_analysis.selection_reasoning` |
| **Формула** | 50% × (длина текста / 240 символов, max 1) + 50% × (есть маркеры «because», «чтобы», «потому что», …) |
| **Типично PMR** | 0.45–0.55 (текст есть, но не всегда идеальные маркеры) |
| **Baseline** | **0.0** (поля нет) |

---

### 4) `alternative_procedures` (отвергнутые альтернативы)

| | |
|---|---|
| **Смысл** | Перечислены ли другие процедуры и **почему** они отклонены |
| **Источник** | `procedural_analysis.alternative_procedures[]` |
| **1.0** | ≥2 элемента с `name` + `rejection_reason` (≥20 символов) |
| **0.5** | 1 такой элемент |
| **0.0** | Нет списка |
| **PMR** | **1.0** во всех прогонах |
| **Baseline** | **0.0** |

---

### 5) `critical_points_density` (плотность критических точек)

| | |
|---|---|
| **Смысл** | Насколько часто в шагах отмечены риски / точки адаптации |
| **Источник** | `solution_steps[].critical_points` |
| **Формула** | Среднее число critical_points на шаг, нормировано: `min(avg / 1.5, 1.0)` |
| **PMR** | **~0.97–1.0** (в шагах обычно 1–2 critical point) |
| **Baseline** | **~0.02–0.06** (поля нет; редкий поиск слов «risk», «критическ» в тексте) |

---

### 6) `adaptation_specificity` (конкретность адаптации)

| | |
|---|---|
| **Смысл** | Есть ли в шагах **конкретные** заметки «если X — сделать Y» |
| **Источник** | `solution_steps[].adaptation_notes` |
| **Зачёт заметки** | ≥30 символов и есть `,` `:` `;` или `—` (признак структурированности) |
| **PMR** | **~0.92–0.98** |
| **Baseline** | **0.0** |

---

### 7) `reflection_actionability` (действенная рефлексия)

| | |
|---|---|
| **Смысл** | Заполнена ли рефлексия и есть ли **действенные** предложения по улучшению |
| **Источник** | `reflection`: effectiveness, limitations, best_use_cases, future_modifications |
| **Формула** | 40% × (доля заполненных ключей) + 60% × (доля future_modifications с глаголами действия: introduce, validate, автоматизир, …) |
| **PMR** | **~0.75–0.85** |
| **Baseline** | **0.0** (блока `reflection` нет) |

---

## A.5. Композит: `procedural_rigor_score`

Итог по задаче — **взвешенная сумма** подметрик (не среднее арифметическое):

| Подметрика | Вес |
|------------|-----|
| procedure_classification | 15% |
| procedure_justification | 15% |
| alternative_procedures | 15% |
| **critical_points_density** | **20%** |
| **adaptation_specificity** | **20%** |
| reflection_actionability | 15% |

**Важно:** `procedural_trace_present` в композит **не входит** — это бинарный индикатор «ответ вообще в формате PMR», он отчётный.

**Пример расчёта (типичная задача PMR):**

```text
classification  1.00 × 0.15 = 0.150
justification   0.49 × 0.15 = 0.074
alternatives    1.00 × 0.15 = 0.150
critical        0.97 × 0.20 = 0.194
adaptation      0.95 × 0.20 = 0.190
reflection      0.82 × 0.15 = 0.123
─────────────────────────────────
procedural_rigor_score      ≈ 0.88
```

**Типичная задача baseline:** все слагаемые ≈ 0 → **rigor ≈ 0.00–0.01**.

## A.7. Парная статистика (PMR vs baseline, procedural)

По **10 задачам** внутри каждого прогона сравниваются пары `(pmr_score, baseline_score)` на одну `task_id`.

### run_01 (типичный прогон)

| Метрика | PMR | Baseline | Δ | p-value | PMR выиграл |
|---------|-----|----------|---|---------|-------------|

## A.8. Связь procedural rigor с сутью PMR (для Discussion)

| Наблюдение | Интерпретация |
|------------|----------------|
| PMR rigor ≈ 0.89, baseline ≈ 0.01 | PMR **по конструкции** производит процедурно-прозрачные ответы; baseline — нет |
| justification только ~0.51 у PMR | Даже PMR не всегда идеально маркирует причинность в `selection_reasoning` — зона улучшения промпта |
| Outcome PMR +7.7% при rigor +88 п.п. | **Качество решения** и **прозрачность процедуры** — разные оси; PMR силён на второй |
| Baseline outcome неплохой (~0.79) | Baseline **не «тупой»** — он решает задачу, но не документирует процедуру |

**Честная формулировка для рецензента:**  
Procedural rigor — это не независимая «объективная истина», а **измеритель соответствия ответа процедурной схеме PMR**. Для сравнения PMR vs baseline это корректно: условия различаются именно требованием процедурной рефлексии. Для сравнения PMR vs PMR-lite (оба с одной схемой) метрику нужно калибровать иначе.

## A.9. Что писать в статье про procedural rigor (готовый блок)

> Мы ввели метрику **procedural rigor** — детерминированный скоринг сохранённого JSON по семи критериям процедурной явности: классификация типа процедуры, обоснование выбора, альтернативы с причинами отказа, плотность critical points в шагах, конкретность adaptation notes и действственность блока reflection. Композит `procedural_rigor_score` — взвешенная сумма (веса 15–20% на компонент). Метрика **не использует gold-разметку** и отделена от outcome-метрик Zarn.  
>  
> На 10 задачах × 3 прогона PMR достиг среднего procedural rigor **0.886** (σ=0.012) против **0.009** (σ=0.005) у structured baseline. Парное сравнение: PMR превосходит baseline на **всех 10 задачах** в каждом прогоне (p<0.001). Baseline сохраняет приемлемый outcome (**0.787**), но практически не генерирует процедурный слой (`procedural_trace_present` = 0). Таким образом, вклад PMR — не только в итоговый балл workflow (+7.7% overall), а в **воспроизводимую процедурную документацию** решения.

## Ограничения

1. **10 задач, 3 прогона** — пилот, не промышленный бенчмарк.
2. **Procedural rigor привязан к схеме PMR** — baseline закономерно проигрывает; это feature сравнения, не bug.
3. **Одна модель** (Qwen3-235B-FP8).
4. **Outcome не всегда значим** по overall при α=0.05 — не преувеличивать p-values.
5. **Не подставлять выдуманные 87/76** — фактические overall: **84.6% / 78.7%**.
